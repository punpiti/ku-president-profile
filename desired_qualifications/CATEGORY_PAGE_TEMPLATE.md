# Template หน้ารายงานรายหมวด

เอกสารนี้สรุปสิ่งที่ทำใน `หมวด 1: ด้านการบริหารทั่วไป` เพื่อใช้เป็นต้นแบบสำหรับทำหมวดถัดไป

## เป้าหมายของหน้ารายหมวด

แต่ละหน้าต้องสกัดจาก PDF แล้วจัดกลุ่มคำตอบตามประเด็นของหมวดนั้น โดยทุกผลวิเคราะห์ต้องย้อนกลับไปดูหลักฐานได้ว่าอยู่ที่ `ข้อไหน`, `หน้าไหน`, และ `ลำดับไหน` ในเอกสาร

โครงสร้างหลักของแต่ละหน้า:

1. ลักษณะทั่วไปตามจำนวนอ้างอิง
2. Sankey: เส้นทางการจัดกลุ่ม
3. ลักษณะพึงประสงค์ของอธิการบดี
4. ความสอดคล้องกับวิสัยทัศน์
5. ข้อกังวล
6. ประเด็นละเอียดอ่อน
7. ประเด็นอื่น
8. ตารางหลักฐานทั้งหมด

## สิ่งที่ทำแล้วในหมวด 1

### Data Pipeline

ไฟล์ที่เกี่ยวข้อง:

- `data/desired_qualifications.txt`: ข้อความจาก PDF แบบ layout
- `data/desired_qualifications.raw.txt`: ข้อความจาก PDF แบบ raw
- `scripts/thai_pdf_cleaner.py`: ทำความสะอาดคำไทยที่เพี้ยนจาก PDF
- `scripts/analyze_pdf.py`: อ่าน PDF text และสร้าง `data/analysis.json`
- `scripts/build_category_report.py`: สร้างข้อมูลรายหมวด
- `data/categories/administration.json`: ข้อมูลหมวด 1
- `categories/administration.html`: หน้าเว็บหมวด 1

หลักการทำความสะอาดข้อความ:

- ใช้ dictionary และ fuzzy pattern เฉพาะคำ ไม่แปลง `า` เป็น `ำ` ทั้งหมด
- ตัวอย่างคำที่แก้: `ด้ำน -> ด้าน`, `บริหำร -> บริหาร`, `ผู้นา -> ผู้นำ`, `จากัด -> จำกัด`
- ระวังหัวข้อแบบ `ด้ำนการบริหำรทั่วไป` เพราะไม่สามารถแก้ด้วยกฎ global ได้

คำสั่ง regenerate:

```bash
python3 scripts/analyze_pdf.py
python3 scripts/build_category_report.py
```

### นิยามการนับ

- `จำนวนอ้างอิงรวม`: ผลรวมของน้ำหนักจากหลักฐานที่เข้าเงื่อนไขในกลุ่มนั้น
- `จาก X ข้อ`: จำนวนรายการหลักฐานจริงที่ถูกจัดเข้าในกลุ่มนั้น
- ถ้าคำตอบใน PDF มีเลขความถี่ในวงเล็บ จะใช้เลขนั้นเป็นน้ำหนัก
- ถ้าไม่มีเลขความถี่ จะนับเป็น 1
- จำนวนอ้างอิงรวมอาจมากกว่าจำนวนข้อ เพราะหนึ่งข้ออาจแทนหลายความเห็นซ้ำ
- Sankey อาจนับซ้ำเชิง flow ได้ เพราะหนึ่งข้ออาจเข้าได้หลาย tag/topic หรือสอดคล้องกับหลายข้อของวิสัยทัศน์

### ความสอดคล้องกับวิสัยทัศน์

ใช้ framework จาก `data/vision_framework.json` แบ่งเป็น 5 กลุ่ม:

1. บริบท
2. บทบาท
3. เป้าหมาย
4. กลยุทธ์
5. แผนสี่ปี

ในหน้าเว็บให้เรียงกลุ่มวิสัยทัศน์จากบนลงล่างตามลำดับนี้เสมอ

สำหรับ Sankey:

- แสดงเฉพาะรายการที่เป็น `ลักษณะพึงประสงค์`
- โครง flow คือ `หมวดนี้ -> tag/topic -> ลักษณะพึงประสงค์ -> วิสัยทัศน์`
- `tag/topic` คือประเด็นที่สกัดจากข้อความก่อน แล้วจึงสรุปเป็น `ลักษณะพึงประสงค์ของอธิการบดี`
- ต้องระบุในคำอธิบายว่า `ลักษณะพึงประสงค์` สกัดมาจากความคิดเห็นของบุคลากร
- ไม่แสดง node `ไม่พบความสอดคล้อง`
- ใช้ `d3-sankey`
- มี fallback renderer เผื่อ CDN โหลดไม่ได้
- มี header คอลัมน์ใน SVG: `หมวดนี้`, `tag/topic`, `ลักษณะพึงประสงค์`, `วิสัยทัศน์`
- เปิดหน้าหมวดให้เลือก default node เป็น `desired_characteristics[0].name` หรือประเด็นลักษณะพึงประสงค์ที่โดดเด่นที่สุดของหมวดนั้น
- หน้า index ใช้ flow เดียวกันแต่คอลัมน์แรกเป็น `หมวด` และ default selection เป็น `เข้าใจนิสิตและพัฒนาคุณภาพชีวิต`
- การคลิก node/link ใช้ highlight ความสัมพันธ์เท่านั้น
- ไม่มีปุ่ม `ดูภาพใหญ่` แล้ว เพราะ fullscreen ทำให้ selection/link rendering ของ Sankey เพี้ยนในบางกรณี

### โครงหน้าและพฤติกรรม UI

ค่าเริ่มต้นตอนเปิดหน้า:

- `ลักษณะทั่วไปตามจำนวนอ้างอิง`: ขยาย
- `Sankey: เส้นทางการจัดกลุ่ม`: ขยาย
- `ลักษณะพึงประสงค์ของอธิการบดี`: ขยาย
- `ความสอดคล้องกับวิสัยทัศน์`: ขยาย
- `ข้อกังวล`: ย่อ
- `ประเด็นละเอียดอ่อน`: ย่อ
- `ประเด็นอื่น`: ย่อ
- `ตารางหลักฐานทั้งหมด`: ย่อ

การ์ดย่อยที่มีเลขอ้างอิง:

- ใน `ลักษณะพึงประสงค์ของอธิการบดี` และ `ความสอดคล้องกับวิสัยทัศน์` ให้ขยายตั้งแต่แรก
- เมื่อขยาย แสดงหลักฐานเริ่มต้น 3 รายการ
- ปุ่มที่ต้องมี: `ยุบทั้งหมด`, `ดูน้อยลง`, `ดูเพิ่ม`, `ขยายทั้งหมด`
- ถ้าหลักฐานมีมากกว่า 3 รายการ ต้องดูเพิ่มทีละ 3 รายการได้

สีและ visual style:

- ใช้ `#006664` เป็นสีหลัก
- เพิ่มสีสดใสใน chart, sankey, badge, card accent
- ไม่ทำให้หน้าเป็นโทนจืดหรือเทาเกินไป
- รักษาความอ่านง่ายของภาษาไทยและตัวเลขอ้างอิง

## ขั้นตอนทำหมวดถัดไป

1. เพิ่ม slug ของหมวดใน `CATEGORY_SLUGS` ใน `scripts/build_category_report.py`
2. ตรวจว่า raw text ของหมวดนั้นเรียงข้อถูกไหม
3. ถ้า raw text ใช้ได้ ให้ตั้ง raw override เฉพาะหมวดนั้น
4. ถ้า raw text ใช้ไม่ได้ ให้ใช้ cleaned layout text หรือเขียน override เฉพาะจุด
5. ปรับ keyword rules สำหรับ topic, desired, concern, sensitive, less relevant ให้เหมาะกับหมวดนั้น
6. regenerate `data/categories/<slug>.json`
7. ทำหน้า `categories/<slug>.html` โดยยึด `categories/administration.html` เป็น template
8. ตรวจ HTML, JS, JSON และลองเปิดหน้าเว็บจริง
9. ตรวจตัวเลข `จำนวนอ้างอิงรวม` เทียบกับตัวอย่างหลักฐานใน PDF

## Checklist ก่อนถือว่าหมวดนั้นเสร็จ

- ชื่อหมวดไม่เพี้ยนจาก PDF
- คำไทยสำคัญถูก clean ก่อนวิเคราะห์
- ทุกการ์ดที่มีจำนวนอ้างอิงสามารถเปิดดูหลักฐานได้
- หลักฐานมี `ข้อ`, `หน้า`, `ลำดับ`, และข้อความอ้างอิง
- `ลักษณะพึงประสงค์ของอธิการบดี` แยกจากข้อกังวลและประเด็นอื่นชัดเจน
- `ประเด็นละเอียดอ่อน` ใช้ชื่อไทย ไม่ใช้คำว่า sensitive ใน UI หลัก
- Sankey ไม่แสดง `ไม่พบความสอดคล้อง`
- Sankey ใช้ flow `หมวดนี้ -> tag/topic -> ลักษณะพึงประสงค์ -> วิสัยทัศน์`
- Sankey มี default selection ของลักษณะพึงประสงค์ที่เด่นสุดในหมวด
- คลิก node หรือเส้นแล้วต้อง highlight ได้
- ไม่มีปุ่ม `ดูภาพใหญ่` หรือ fullscreen Sankey
- วิสัยทัศน์เรียงตาม `บริบท`, `บทบาท`, `เป้าหมาย`, `กลยุทธ์`, `แผนสี่ปี`
- สีหลักยังเป็น `#006664`
- หน้าแรกยังไม่เสีย และ link ไปหน้าหมวดถูกต้อง

## ข้อจำกัดที่ต้องจำไว้

- การจัดกลุ่มตอนนี้ยังอิง keyword matching เป็นหลัก ไม่ใช่ semantic model เต็มรูปแบบ
- ตัวเลข Sankey เป็น flow count ไม่ใช่กลุ่มที่ mutually exclusive แต่ต้องกระจายน้ำหนักของ evidence เพื่อไม่ให้ยอดพองเมื่อหนึ่งข้อมีหลาย topic, desired tag, หรือ vision alignment
- PDF ภาษาไทยมีข้อความเพี้ยน ต้องตรวจคำสำคัญของแต่ละหมวดก่อนสรุป
- หน้า HTML ตอนนี้ยังเป็น template แบบ copy ได้ แต่ควร refactor เป็น generic renderer ถ้าจะทำครบ 7 หมวด

## งานที่ควรพิจารณาก่อนรันยาวครบ 7 หมวด

- ทำ `categories/template.html` หรือ renderer กลางที่รับ slug จาก query/path
- แยก taxonomy ของแต่ละหมวดเป็น config JSON
- เพิ่มรายงานตรวจคำเพี้ยนที่พบบ่อยต่อหมวด
- เพิ่ม export ตารางหลักฐานเป็น CSV
- เพิ่ม summary ว่าประเด็นไหนมีน้ำหนักสูง แต่หลักฐานกระจุกอยู่ในหน้า/ข้อเดียวกันเกินไป
